These reads are not optional. The patterns in these files are NOT documented in the
.md files — they live only in the code. Skipping them causes auth failures and broken
UI that look like bugs but are actually pattern deviations.

Specific rules derived from past failures:
- NEVER use raw fetch() for authenticated endpoints — always api.stream() or api.get()
- api.get() returns the parsed JSON body directly — there is no .data wrapper
- Agent run history rows have shape { id, status, result, run_at } where result is
  the JSONB object containing { summary, data, ... } — access as run.result?.summary
- SSE result events have shape { type: 'result', data: { summary, data, ... } } —
  access as ev.data?.summary, not ev.data directly
- persistRun() saves status as 'complete' (not 'success') and 'error' (not 'failure')
- Always spread { startDate, endDate } into the agentOrchestrator context object

## What you are building in this session

A new agent called `high-intent-advisor` and its supporting infrastructure.

This agent runs daily, analyses data across Google Ads, GA4, and the WordPress CRM, 
and produces specific evidence-backed suggestions for driving high-intent customers 
to the site who will make an enquiry. It tracks its own suggestions over time, 
measures outcomes, and uses that history to improve future suggestions.

## Step 1 — DB migration (db.js)

Add the following table to the idempotent schema in db.js using the 
existing ALTER TABLE ADD COLUMN IF NOT EXISTS pattern.
Do not create a new migration file — add it inline with the existing schema init.

```sql
CREATE TABLE IF NOT EXISTS agent_suggestions (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id           INTEGER REFERENCES organizations(id),
  run_id           UUID,
  slug             TEXT NOT NULL DEFAULT 'high-intent-advisor',
  category         TEXT NOT NULL,
  priority         TEXT NOT NULL DEFAULT 'medium',
  suggestion_text  TEXT NOT NULL,
  rationale        TEXT NOT NULL,
  status           TEXT NOT NULL DEFAULT 'pending',
  baseline_metrics JSONB DEFAULT '{}',
  outcome_metrics  JSONB DEFAULT '{}',
  outcome_notes    TEXT,
  acted_on_at      TIMESTAMPTZ,
  reviewed_at      TIMESTAMPTZ,
  created_at       TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_agent_suggestions_org_status 
  ON agent_suggestions(org_id, status, created_at DESC);

CREATE INDEX IF NOT EXISTS idx_agent_suggestions_run 
  ON agent_suggestions(run_id);
```

Valid values:
- category: 'keyword' | 'budget' | 'landing_page' | 'audience' | 'search_term' | 'device' | 'scheduling'
- priority: 'high' | 'medium' | 'low'
- status: 'pending' | 'acted_on' | 'monitoring' | 'dismissed'

## Step 2 — Two new platform tools in platform.js MCP server

Add these two tools to the existing platform.js MCP server 
(server/mcp-servers/platform.js). Do not create a new MCP server.
Follow the exact tool definition pattern already used in that file.

### get_pending_suggestions
- Description: Returns all suggestions for this org with status 'pending' or 
  'monitoring', ordered by priority (high first) then created_at DESC.
- Input schema: { type: 'object', properties: {}, required: [] }
- Returns: Array of suggestion rows including id, category, priority, 
  suggestion_text, rationale, status, baseline_metrics, outcome_notes, 
  created_at, reviewed_at
- SQL: SELECT * FROM agent_suggestions 
  WHERE org_id = $1 AND status IN ('pending', 'monitoring') 
  ORDER BY CASE priority WHEN 'high' THEN 1 WHEN 'medium' THEN 2 ELSE 3 END,
  created_at DESC

### update_suggestion_outcome
- Description: Updates outcome_metrics, outcome_notes, and reviewed_at on a 
  suggestion row. Called by the agent during its review phase.
- Input schema: 
  {
    type: 'object',
    properties: {
      suggestion_id: { type: 'string', description: 'UUID of the suggestion row' },
      outcome_metrics: { type: 'object', description: 'Current metric values for comparison against baseline' },
      outcome_notes: { type: 'string', description: 'Agent assessment of whether this suggestion moved the needle' },
      status: { type: 'string', description: 'Updated status if appropriate: monitoring | pending' }
    },
    required: ['suggestion_id', 'outcome_metrics', 'outcome_notes']
  }
- Returns: { updated: true, suggestion_id }
- SQL: UPDATE agent_suggestions 
  SET outcome_metrics = $1, outcome_notes = $2, reviewed_at = now(),
  status = COALESCE($3, status)
  WHERE id = $4 AND org_id = $5
- orgId must come from the MCP server session context, not from the tool input.
  The tool must validate that the suggestion belongs to this org before updating.

After adding these tools, increment the platform.js server version.
Update MCP-SERVERS.md to add both tools to the platform.js table.

  Google Ads conversation agent
- emit() progress updates at each phase boundary:
  "Phase 1 complete — reviewed N pending suggestions"
  "Phase 2 complete — data gathered across all sources"  
  "Phase 3 complete — N suggestions generated"
- After AgentOrchestrator completes, parse the agent output to extract 
  structured suggestions. Each suggestion must have all required fields.

  how many pending suggestions were reviewed, what the overall data picture 
  showed, how many new suggestions were generated and at what priorities.

Parsing the agent output into structured suggestions:
The agent will produce a text response. You need to extract suggestions from it.
Prompt the agent to wrap each suggestion in a structured format you can parse.
Add this instruction to the end of Block 3:

"Format each suggestion as a JSON block wrapped in <suggestion> tags:
<suggestion>
{
  "category": "...",
  "priority": "...", 
  "suggestion_text": "...",
  "rationale": "...",
  "baseline_metrics": {}
}
</suggestion>

Wrap each suggestion individually. Plain text analysis and phase summaries 
go outside the tags."

Then in index.js, parse with:
const suggestionMatches = result.matchAll(/<suggestion>([\s\S]*?)<\/suggestion>/g);

Handle JSON parse errors per suggestion gracefully — 
a malformed suggestion block should not fail the entire run.
