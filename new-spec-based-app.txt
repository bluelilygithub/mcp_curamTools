## Agent: Spec Validator

### What it does
Accepts a PDF hydraulic services calculation document, extracts 
all quantitative claims (pipe sizes, flow rates, velocities, 
pressure drops, pressure budgets), runs deterministic 
Python-based calculations to verify each claim, and returns 
a structured finding set for human review. A compliance 
certificate is generated only when all findings have been 
reviewed and every finding is approved — rejected findings 
block the certificate permanently; the engineer must resubmit 
a corrected document.

This agent is distinct from demo-document-analyzer. The 
document analyser is broad-scope contract review. The Spec 
Validator is narrow and numerical — it is specifically for 
engineering calculation documents containing quantitative 
hydraulic specifications. The follow-up Q&A is scoped 
exclusively to the discrepancies found in the current run 
and direct remediation questions arising from them (e.g. 
"what pipe size would bring segment CW-04 into compliance?"). 
The system prompt and the user message closing instruction 
must both enforce this boundary using the same dual-layer 
pattern documented in DECISIONS.md under 
"Document Analyzer Follow-up Q&A — Scope Restriction".

## Three-stage pipeline

Stage 1 — Extraction (Claude):
Claude reads the uploaded PDF and extracts every quantitative 
claim into a structured JSON object. Each extracted item must 
include: segment reference, description, parameter name, 
stated value, unit, and source location in the document 
(page number and context string). Claude does not perform 
any calculations at this stage — extraction only.

Stage 2 — Validation (Python subprocess):
A Python script receives the Stage 1 JSON and runs 
deterministic calculations for each extracted item. 

The Python layer must:
- Set up and use appropriate libraries for the calculations. 
  Choose libraries that are reliable, well-maintained, and 
  suited to deterministic hydraulic calculations. Do not use 
  AI or probabilistic methods in this layer.
- Implement Hazen-Williams pressure drop calculation
- Implement Darcy-Weisbach as an alternative where 
  friction factor is specified
- Implement continuity equation velocity check (Q = V·A)
- Implement static pressure budget summation and residual 
  pressure check against AS/NZS 3500.1 minimum fixture 
  pressures
- Implement Reynolds number and flow regime classification
- Show full working for every calculation — intermediate 
  values, formula used, inputs, result — not just pass/fail
- Return a structured JSON result for each check: 
  stated_value, calculated_value, unit, formula_used, 
  working (step-by-step), status (PASS/FAIL/WARNING), 
  tolerance_applied, standard_reference

The Python subprocess must be called via execFileAsync 
(not exec, not spawn with shell:true). Never use child_process 
shell execution for the calculation layer.

Stage 3 — Synthesis (Claude):
Claude receives the Stage 2 JSON and produces reviewer-ready 
findings. Each finding must include: a plain-language 
explanation of what the discrepancy means in practice 
(not just what the numbers are), the likely cause of the 
error, and the recommended remediation. The synthesis must 
not introduce any new calculations — it interprets and 
communicates the Python output only.

## Finding classification

Two finding types, matching the document analyser pattern:

Deterministic findings (from Python Stage 2):
- Confidence always 1.0
- Status: PASS, FAIL, or WARNING
- FAIL = value outside code limit (e.g. velocity > 3.0 m/s)
- WARNING = value within limits but anomalous 
  (e.g. margin < 5 kPa, velocity < 0.5 m/s)

Probabilistic findings (from Claude Stage 3):
- Confidence 0.0–1.0
- Flags patterns that require engineering judgment 
  (e.g. "pipe schedule does not include fittings allowance", 
  "diversity factor not stated — assumed value used")

Cross-stage overlap detection: implement the same overlap 
flagging as document analyser. Overlapping findings require 
a comment before approval (enforced server-side in the 
PATCH endpoint).

Low confidence (< 0.7) approved findings require a comment. 
Rejected findings cannot be approved — they permanently 
block the certificate. The engineer must resubmit a corrected 
document for a new run.

## HITL review flow

Follow the document analyser pattern exactly:
- Findings stored in agent_runs.result.data
- Review state updated via 
  PATCH /api/demo/runs/:runId/review/:findingId
- Uses jsonb_set — no separate table
- Syncs finding across all three arrays atomically

Server-side enforcement:
- status must be approved | rejected | resubmit
- low confidence + approved with no comment → 422
- cross-stage overlap + approved with no comment → 422
- rejected finding → certificate permanently blocked 
  (no override path)

## Compliance certificate

- Built client-side as HTML string, exported via exportPdf
- Gated on: all findings reviewed AND all findings approved 
  (zero rejected, zero pending)
- Rejected findings permanently block the certificate — 
  this is by design and must not have an override path
- Certificate content must include: document name, run date, 
  model used, total findings, pass/fail/warning breakdown, 
  Python library versions used for calculations, 
  reviewer name and timestamp for each finding, 
  statement that deterministic calculations were performed 
  by Python (not AI), and the full calculation working 
  for each FAIL finding

## Constraints — must not violate

- org_id always from req.user.orgId — never from a 
  request param
- Always pass customProviders to getProvider
- Use ?? fallback at call site for nullable admin config — 
  not JS default params
- PDF export: exportService.js only
- https.request for outbound HTTP — never fetch()
- execFileAsync for Python subprocess — never shell exec
- Extraction privacy applied post-AI pre-return via 
  AgentConfigService.getExtractionPrivacySettings(orgId)
- No hardcoded model default — throw a clear error if 
  no model is configured
- Use shared server/utils/sanitize.js for input scanning — 
  scan filename and any custom prompt text, NOT the 
  document content
